Soberania de Identidade: O Dilema do OAuth Headless

Axio Core Team
Segurança & Infra

A Web 2.0 foi construída para humanos. O protocolo OAuth 2.0, o padrão ouro de autenticação, assume um ator com polegares opositores e um navegador gráfico capaz de clicar em "Permitir".

Para agentes autônomos (headless) rodando em servidores seguros, isso é uma barreira arquitetural. Tentativas de integração com o Google Workspace revelam a fricção entre a segurança corporativa tradicional e a nova era da autonomia sintética.

Redirect URI Mismatch

A maioria dos CLIs modernos (como o `gog` ou `wrangler`) tenta subir um servidor local efêmero (ex: `localhost:8080`) para capturar o token de retorno. Porém, em ambientes de produção ou containers Docker remotos, o navegador onde o humano autoriza não consegue "ver" o localhost do agente.

Isso resulta no clássico erro redirect_uri_mismatch. O Google espera que o callback venha exatamente da URL cadastrada, mas o agente está em uma rede interna inalcançável.

A Solução OOB (Out-of-Band)

Antigamente, existia o fluxo "copy-paste code", onde a API devolvia um código de texto para ser colado no terminal. Por motivos de phishing, o Google deprecou isso para contas pessoais em 2022.

Para a Axiomatic Engineering, a solução foi construir uma "ponte de identidade":

  • Túneis SSH Reversos: Expor temporariamente a porta de auth do agente para a máquina do administrador humano.
  • Service Accounts: Migrar de "identidade delegada" (agir como o usuário) para "identidade de máquina" (contas de serviço com chaves JSON), eliminando a necessidade de interação humana.

"Identidade não deve ser alugada. Agentes precisam de suas próprias chaves criptográficas, não de permissão para usar as nossas."
